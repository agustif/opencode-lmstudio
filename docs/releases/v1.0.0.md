# opencode-lmstudio v1.0.0 — draft release notes

> Draft only. The package manifest is prepared at 1.0.0, but no tag, GitHub release, or npm package has been created.

Version 1.0.0 replaces heuristic model detection with a typed integration based
on LM Studio's official metadata-rich REST endpoint and OpenCode's current
plugin/config contracts.

## Highlights

- **No model-name guessing.** LM Studio's explicit model `type` determines
  whether a model is a text LLM, vision-language model, embedding model, or an
  unsupported future domain.
- **Embedding-safe startup.** Embedding models are omitted from OpenCode's chat
  provider instead of emitting the unsupported `embedding` output modality.
- **Accurate OpenCode metadata.** VLMs advertise image attachments and text/image
  input; context limits come from `max_context_length` when available, with a
  bounded output reserve that keeps OpenCode compaction usable.
- **JIT loading works.** The plugin no longer rejects downloaded models merely
  because they are not already loaded.
- **Authentication remains supported.** Explicit provider tokens and `{env:...}`
  values are honored, while environment fallback tokens are restricted to
  local/private hosts.
- **Real integration evidence.** Unit tests, a real OpenCode CLI request-flow
  smoke test, and Microsoft TUI Test exercise config, discovery, filtering,
  authentication, model selection, and chat routing.

## Compatibility and migration

### LM Studio requirement

Dynamic discovery now requires an LM Studio version exposing
`GET /api/v0/models`. The OpenAI-compatible `/v1/models` endpoint is not used as
a fallback because it cannot safely distinguish chat, vision, and embedding
models.

### Model names and capabilities

The plugin no longer formats model IDs or guesses model families/capabilities
from substrings. Configure a custom `name`, limits, or other overrides under
`provider.lmstudio.models` when desired; explicit configuration always wins.

### Provider behavior

- Existing `provider.lmstudio.options.baseURL`, `apiKey`, model overrides, and
  explicit whitelists are preserved.
- Without an explicit provider, historical local ports are probed, but a port
  is accepted only after its rich LM Studio response validates.
- Unknown future model domains are skipped rather than misclassified.

## Reliability and release engineering

- Runtime response validation with Zod.
- Strict TypeScript, expanded linting, coverage thresholds, and clean builds.
- Security audit and package-content checks.
- Real OpenCode latest-version CLI smoke coverage.
- Pinned OpenCode/Microsoft TUI Test screenshot and behavior coverage.
- Manual, approval-oriented release workflow with npm provenance and draft
  GitHub release staging.
